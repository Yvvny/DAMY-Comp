# DAMY Proofing Workflow

This document explains how the Proofing part of DAMYComp works from the user's point of view and from the code's point of view.

## Main Entry Points

The normal app starts from `run_folder_manager.py`, which calls `folder_manager.app.run_app()`. Inside the main DAMYComp window, the Proofing workspace is embedded from:

- `folder_manager/proofing_online/ui/views/main_window.py`

The proofing workspace uses these main support modules:

- `folder_manager/proofing_online/workflow_config.py` - stage names and stage numbers.
- `folder_manager/proofing_online/workflow_services.py` - shared stage logic for sorting, upload naming, PDF discovery, and school email drafts.
- `folder_manager/proofing_online/path_resolver.py` - resolves job folders, runtime folders, PDF folders, and nested stage folders.
- `folder_manager/proof_sorter/sorter.py` - sorts proof images into student folders.
- `folder_manager/photodeck_upload/` - uploads sorted galleries to PhotoDeck and generates student PDFs.
- `folder_manager/sms/child_info.py` - prepares parent delivery rows, preview assets, MMS, and parent email delivery.
- `folder_manager/proofing_online/order_import/` - imports PhotoDeck paid orders from Gmail and copies ordered assets.

## Proofing Stages

Proofing stages are defined in `folder_manager/proofing_online/workflow_config.py`.

| Stage | Column name | Purpose |
| --- | --- | --- |
| 1 | `1. Proofs` | Initial proof job stage. |
| 2 | `2. Sort` | Sort raw proof images into per-student proof folders. |
| 3 | `3. Upload & PDFs` | Upload sorted proofs to PhotoDeck, generate PDFs, and prepare parent delivery assets. |
| 4 | `4. School / Parent Delivery` | Send proof PDFs to the school and send parent proof links/previews. |
| 5 | `5. Finished` | Job is finished after school/parent proof delivery work. |
| 6 | `6. Edit` | Paid PhotoDeck order assets are ready for edit work. |
| 7 | `7. Print` | Selected original paid-order assets are ready for print/package work. |
| 8 | `8. Package` | Package stage. |
| 9 | `9. Deliver` | Final delivery stage. |

The UI can also recognize older aliases for some columns, for example old stage 3 names like `3. Photodeck Upload`, `3. PDF Packets`, or `3. Upload/PDF Packets`.

## Folder And Path Resolution

The Proofing workspace is driven by database rows plus matching folders on disk. The path resolver tries to find folders by `disk_name` under the configured base directory. It also checks `cancel` folders and some nested stage folders.

Important path behavior:

- The default base is usually `T:\DAMY`, or the value passed into the embedded Proofing widget.
- Runtime files go under `_workflow_runtime/proofing`.
- Stage 3 PDF discovery checks a saved DB link first, then `job_folder/PDFs`, then `stage3_work_root/PDFs`.
- `Parent Delivery Assets` folders are ignored when collecting school PDF attachments.

## Stage 2: Sort

Stage 2 takes an unsorted source folder and creates a final sorted proof output folder.

The main flow is:

1. User selects a folder in `2. Sort`.
2. User chooses the source folder to sort.
3. The app builds a `Stage2SortPlan`.
4. Sorting runs into a temporary folder first.
5. If sorting succeeds, the app replaces the old sorted output.
6. If the source folder is different from the job folder, the app deletes the old source after the new output succeeds.
7. If files were sorted, the job is automatically moved to stage 3.

The output folder name is generated by `stage2_proof_output_name()`. For a job name containing a picture day ID such as `P02262835`, the output becomes a name like:

```text
260324 TEST SCHOOL Proof P02262835
```

The sorter groups images by the trailing parenthesized student part in the filename. Student folder names use the proof password format:

```text
Student Name_1234_
```

Passwords are handled in `folder_manager/proofing_online/passwords.py`. A trailing four-digit password can be extracted from sorted student folders, and the same password is later used for PhotoDeck/PDF behavior.

The sorter can create a `#COVER.jpg` in each student folder. Stage 2 calls the sorter with:

- `generic_password=os.getenv("DAMY_PROOFING_GENERIC_PASSWORD", "")`
- `make_covers=True`
- `copy_files=True`
- `auto_password=False`

This means Stage 2 copies images into the proof output instead of moving the image files during the sort operation itself.

## Stage 3: Upload & PDFs

Stage 3 is the combined PhotoDeck upload, PDF generation, and parent delivery preparation step.

The main flow is:

1. User selects a folder in `3. Upload & PDFs`.
2. User clicks `Upload to PhotoDeck`.
3. User chooses the folder to upload. The default is the job's `sorted` folder when it exists and is not empty.
4. The app builds a `Stage3UploadPlan`.
5. The app uploads the selected folder through `folder_manager.photodeck_upload.workflow.run_bulk_upload_workflow`.
6. After upload succeeds, the app generates PDFs through `folder_manager.photodeck_upload.generate_student_pdfs.run_generate_student_pdfs`.
7. The PDF folder link is saved to the DB as a workflow asset.
8. The app tries to prepare parent delivery rows and preview assets.
9. If upload succeeded, the job is automatically moved to stage 4.

The picture day ID used for upload is inferred from the job name. If the job name contains `P########` or `H########`, that value is used. Otherwise, the app falls back to a leading date token or today's date.

The PhotoDeck gallery name comes from the selected upload folder or job folder with the stage prefix removed.

Stage 3 also writes a small runtime manifest so later paid-order import can find the proof upload source for the job:

```text
_workflow_runtime/proofing/photodeck_upload_sources.json
```

## Stage 4: School / Parent Delivery

Stage 4 has two separate delivery flows.

### School Delivery

School delivery sends the generated proof PDFs to the school contact.

The main flow is:

1. User selects a folder in `4. School / Parent Delivery`.
2. The app finds the school contact email from the workflow row.
3. If no valid email exists, the user must enter one in School Contact.
4. The app collects PDFs from the saved Stage 3 PDF link or the job PDF folders.
5. The user reviews the Gmail dialog.
6. The app sends one email with the PDFs attached.
7. The DB row is marked with school email sent status.

The default email subject is:

```text
<School Name> - Proofs
```

Sending the school email does not move the job to another stage by itself.

### Parent Delivery

Parent delivery uses the PDFs and parent contact information to prepare child rows, cloud preview links, MMS, and parent email delivery.

Parent delivery requires:

- Stage 3 PDFs to exist.
- Cloudflare R2 settings when preview assets need to be published.
- Parent contact data saved or imported through the parent delivery dialog.

The parent delivery dialog is implemented outside the main proofing view, mainly in:

- `folder_manager/ui/child_mms_dialog.py`
- `folder_manager/sms/child_info.py`
- `folder_manager/sms/cloudflare_r2.py`
- `folder_manager/sms/twilio_client.py`

## Paid PhotoDeck Order Import

The `Import Paid Orders` button runs the paid-order importer in a subprocess through:

```text
folder_manager.proofing_online.order_import.paid_runner
```

The runner can also be reached from `run_folder_manager.py` with:

```text
--photodeck-paid-import
```

The paid-order importer:

1. Connects to Gmail.
2. Looks for PhotoDeck payment receipt messages.
3. Extracts picture day IDs from messages.
4. Matches the receipt to an active proofing job by PID.
5. Parses order numbers, ordered photo identifiers, packages, backgrounds, add-ons, and quantities.
6. Finds original images and matching proof images.
7. Copies assets into the job's `Orders` folder.
8. Records copied assets in `proofing_paid_order_assets`.
9. Records an order import event in `workflow_events`.
10. Optionally moves processed Gmail messages to the imported label.

The output folders used by paid-order import are:

```text
<job folder>/Orders
<job folder>/Orders/Order PDFS
```

Original and proof image matching is done by exact IDs parsed from the PhotoDeck receipt. Proof source lookup can use the Stage 3 upload source manifest, which is why Stage 3 records the upload source.

## Stage 6 And Stage 7 Paid Assets

Paid-order image assets are tracked in the `proofing_paid_order_assets` table.

Important columns include:

- `workflow_run_id`
- `disk_name`
- `pid`
- `order_no`
- `message_id`
- `asset_type`
- `original_id`
- `proof_id`
- `path`
- `source_path`
- `order_pdf_path`
- `package`
- `addons`
- `background`
- `quantity`
- `asset_status`

The `asset_status` value is one of:

- `stage6`
- `stage7`
- `archived`

Stage 6 shows imported paid-order assets waiting for edit/print preparation. Stage 7 is focused on original assets selected for print. The UI can move selected original assets from stage 6 to stage 7 by updating the DB status for the asset group.

## Move To Edit / Returned From Edit

The Proofing UI has additional controls for edit handling.

`Move to Edit`:

- Requires user confirmation.
- Requires the password `LIZA`.
- Collects image files from order/import folders.
- Clears and rebuilds a runtime `Send To Edit` folder.
- Updates the related workflow rows to stage 6.

`Returned From Edit`:

- User selects a folder of edited images.
- The app matches edited image base names back to edit folders.
- Edited images are copied back into the DAMY job folders with ` Edited` added to the filename.
- Finished job folders remain in stage 5.

## Database State

Proofing uses the shared workflow DB tables plus proofing-specific paid-order asset tracking.

The main shared workflow records are in `workflow_runs`, with `workflow_domain='proofing'`.

Important proofing-specific DB behavior:

- Stage moves use `update_domain_stage()`.
- New rows use `upsert_into_domain()`.
- PDF links are saved as workflow assets.
- School email status is stored on the workflow row.
- Paid PhotoDeck imported assets are stored in `proofing_paid_order_assets`.
- Imported paid orders are also recorded as `workflow_events` with event type `order_imported`.

## Deployment Notes

Proofing is included in the main PyInstaller build through `DAMYComp.spec` and `build_folder_manager.bat`.

There is also a proofing deployment script:

```text
Deploy Proofing EXE To T DAMYComp.bat
```

That script publishes a versioned Proofing EXE release under the configured Proofing share root, defaulting to:

```text
T:\DAMYComp_Proofing
```

## Short End-To-End Summary

The normal proofing flow is:

1. Add or find a Proofing job in the board.
2. Stage 2 sorts source images into passworded student proof folders.
3. Stage 3 uploads the sorted folders to PhotoDeck, generates PDFs, saves the PDF path, and prepares parent delivery.
4. Stage 4 sends the school PDF packet and handles parent proof delivery.
5. The job can finish at stage 5.
6. Later, paid PhotoDeck orders are imported from Gmail.
7. Imported paid-order assets go to stage 6 for edit work.
8. Selected original assets can move to stage 7 for print, then continue through package and delivery.
